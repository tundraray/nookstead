## 1. Цель

Этот документ задает детальный алгоритм:

- когда после покраски клетки нужно **фактически обновлять** результат пересчета соседних клеток;
- когда пересчет соседа технически выполняется, но его результат **не изменяет** видимый вывод;
- как классифицировать ребра между материалами для предсказуемого визуального результата.

Документ является **дополнением** к основному Design Doc (design-015) и ADR-0011. Он не заменяет их, а уточняет поведение внутри существующей архитектуры RetileEngine.

### 1.1. Принцип совместимости

Этот документ **не меняет** следующие архитектурные решения:

| Решение | Источник | Статус |
|---------|----------|--------|
| Chebyshev R=2 dirty propagation | ADR-0011 Decision 2, Design Doc AC6 | Сохраняется без изменений |
| RetileEngine.expandDirtySet (до 25 клеток на покраску) | retile-engine.ts | Сохраняется без изменений |
| Reverse-pair fallback (ADR Decision 7) | ADR-0011 Decision 7 | Полноценный механизм |
| S1/S2 conflict resolution | ADR-0011 Decision 6 | Работает на всех ребрах |
| EdgeResolver (3 варианта: both-valid / one-valid / neither-valid) | edge-resolver.ts | Сохраняется без изменений |
| Порог 50% для full-pass | retile-engine.ts expandDirtySet | Сохраняется без изменений |

## 2. Ключевая идея

### 2.1. Два уровня: зона пересчета и визуальная стабильность

Архитектура RetileEngine оперирует **зоной кандидатов на пересчет** (dirty set) -- множеством клеток, для которых движок пересчитывает тайлсет, маску и фрейм. Эта зона определяется **геометрически** (Chebyshev R=2) и не подлежит сужению.

Политика, описанная в этом документе, работает на **другом уровне**. Она не фильтрует dirty set, а отвечает на вопрос: **для каких пересчитанных клеток результат будет визуально осмысленным?** Это знание полезно для:

1. **Понимания поведения**: разработчик может предсказать, какие клетки реально изменят свой визуал после покраски.
2. **Диагностики**: если сосед не обновился визуально, это может быть ожидаемым поведением, а не багом.
3. **Будущей оптимизации**: если в будущем потребуется уменьшить dirty set для производительности, классификация ребер подскажет, какие клетки можно безопасно исключить.
4. **Тестирования**: для какой пары ожидать визуальное изменение соседа, а для какой нет.

### 2.2. Общий принцип

Все клетки в dirty set (R=2 Chebyshev) технически пересчитываются. Compute-фаза детерминирована: одинаковый полный вход compute-фазы дает одинаковый кандидатный результат.

Но фиксация визуального изменения для соседей должна быть управляемой:

1. Для устойчивых классов ребра (C1/C2/C3) изменение кардинального соседа считается ожидаемым и фиксируется.
2. Для неустойчивых классов (C4/C5) изменение кардинального соседа не фиксируется как обязательное поведение.
3. Диагональный сосед фиксируется только в устойчивом углу: обе прилегающие кардинали должны быть стабильными (`C1/C2/C3`).
4. Центр покраски всегда обновляется.

Таким образом документ разделяет:

1. вычислительный пересчет (engine-level);
2. политику визуальной стабилизации соседей (render-level policy).

Важно: в этом документе "фиксируется" означает "клетка входит в commit-набор". Это не эквивалентно гарантированной смене номера `frame`: после commit кадр может совпасть с предыдущим.

## 3. Термины

1. **Центр** -- клетка, материал которой пользователь только что изменил. В batch-триггерах (`fill/rectangle`) центр трактуется как множество измененных клеток (`centers`).
2. **Сосед (кардинальный)** -- в рамках классификации это кардинальный сосед центра (N/E/S/W). Для него напрямую применяется классификация C1-C5.
3. **Сосед (диагональный)** -- не классифицируется как отдельное ребро. Фиксируется только если обе прилегающие к нему кардинали стабильны (`C1/C2/C3`).
4. **Dirty set** -- множество клеток, пересчитываемых RetileEngine после покраски. Определяется геометрически: Chebyshev R=2 вокруг каждой покрашенной клетки. При >50% карты -- full-pass.
5. **Прямая пара** (direct pair) -- существует тайлсет `A_B` (`fromMaterialKey=A, toMaterialKey=B`).
6. **Обратная пара** (reverse pair) -- прямой пары `A_B` нет, но есть `B_A`. Используется с обратной ориентацией маски. Является **полноценным** механизмом по ADR-0011 Decision 7.
7. **Маршрут** (route) -- результат `nextHop(A, B)` из BFS routing table. Может возвращать промежуточный материал-мост M или null (нет пути). Таблица строится по `compatGraph` (совместимость недиректед): ребро `A <-> B` существует, если в наборе есть хотя бы один переход `A_B` или `B_A`.
8. **Владелец ребра** (edge owner) -- результат `resolveEdge()`: сторона границы, которая определяет тайлсет перехода на этом ребре. Используется в S1/S2 conflict resolution.
9. **Resolved pair** -- результат `registry.resolvePair(fg, bg)`: возвращает `{ tilesetKey, orientation: 'direct' | 'reverse' }` или null.

### 3.1. Краткий глоссарий (быстрый lookup)

| Термин | Что это простыми словами |
|---|---|
| `compute-phase` | Пересчет кандидатных `tileset/frame` для всех клеток dirty set без фильтрации |
| `commit-phase` | Выбор, какие из кандидатных результатов реально записываются в visual/cache |
| `mandatoryCommitSet` | Клетки, которые обязательно фиксируются на paint (`center`, стабильные кардинали, стабильные диагонали) |
| `optionalCommitSet` | Остальные клетки dirty set; на paint обычно сохраняют прошлый визуал, если он уже есть |
| `owner-side` | Правило: открытие перехода по ребру делает только владелец ребра |
| `bridge` (`M`) | Промежуточный материал маршрута между двумя FG (пример: `deep_water -> water -> grass`) |
| `base-mode` | Режим без transition-bg (`bg=''`), фрейм по связности собственного FG |
| `transition-mode` | Режим с выбранным `bg`, где кардинали открываются/закрываются по owner-side контракту |
| `ring-solid override` | Спец-правило только для isolated C2: `center=47`, первый пояс принудительно `1` |

## 4. Классификация ребер

### 4.1. Зачем нужна классификация

EdgeResolver (edge-resolver.ts) имеет три исхода: both-valid / one-valid / neither-valid. Классификация ребер **расширяет** этот набор дополнительной информацией о **качестве стыка**, что позволяет предсказать визуальный результат.

Сама классификация не меняет расчеты маршрутизации/маски. Она используется policy-layer для решения, фиксировать ли визуальное изменение соседа, и является основой для будущих оптимизаций.

### 4.2. Исходные данные для классификации ребра A <-> B

Для каждого кардинального ребра между материалами A и B вычисляются:

```
hopA = nextHop(A, B)     -- маршрут A в сторону B
hopB = nextHop(B, A)     -- маршрут B в сторону A
pairA = resolvePair(A, hopA)  -- тайлсет для A-стороны (direct/reverse/null)
pairB = resolvePair(B, hopB)  -- тайлсет для B-стороны (direct/reverse/null)
```

Контракт router для этой классификации: `nextHop` работает по графу совместимости (а не только по direct-ребрам render-графа), поэтому reverse-пара участвует в достижимости пути.

### 4.3. Пять классов ребер

| Класс | Условие | Визуальная стабильность | Пример |
|-------|---------|------------------------|--------|
| **C1: Одноматериальный** | A === B | Всегда стабилен | grass <-> grass |
| **C2: Симметричный прямой** | pairA.orientation === 'direct' И pairB.orientation === 'direct' И hopA === B И hopB === A | Высокая | grass <-> water (оба направления существуют как прямые тайлсеты) |
| **C3: Устойчивый с мостом** | pairA.orientation === 'direct' И pairB.orientation === 'direct' И hopA === hopB И hopA !== A И hopA !== B | Высокая | deep_water <-> grass через мост water (оба прямые) |
| **C4: Смешанный валидный** | pairA !== null И pairB !== null, но не C2/C3 (хотя бы одна сторона reverse, или мосты разные) | Средняя/низкая | deep_water <-> water (direct + reverse) |
| **C5: Частичный или невалидный** | hopA === null ИЛИ hopB === null ИЛИ pairA === null ИЛИ pairB === null | Низкая | Гипотетическая несвязная пара |

### 4.4. Связь с EdgeResolver

| EdgeResolver result | Возможные классы ребра |
|--------------------|----------------------|
| both-valid | C2, C3, C4 |
| one-valid | C5 |
| neither-valid | C5 |
| null (same material) | C1 |

Классификация ребер -- это надстройка над EdgeResolver. Для визуальной стабилизации используется отдельное решение:

1. C1/C2/C3 -> кардинальный сосед считается пригодным к фиксации изменения.
2. C4/C5 -> кардинальный сосед не считается обязательным кандидатом на фиксацию изменения.
3. Диагональный сосед фиксируется только при двух стабильных прилегающих кардиналях (`C1/C2/C3`).

### 4.5. Глоссарий классов C1-C5 (простыми словами)

| Класс | Что означает | Что это значит для commit соседей |
|---|---|---|
| `C1` | Оба материала одинаковые (`A === B`) | Сосед стабильный, commit ожидаем |
| `C2` | Оба направления прямые и симметричные (`A_B` и `B_A`) | Сосед commit обязателен; isolated-center ring-solid допустим |
| `C3` | Обе стороны прямые через один и тот же мост `M` | Сосед commit обязателен; первый пояс считается по коннекторам |
| `C4` | Обе стороны валидны, но есть reverse или мосты разные | Сосед не обязателен к commit (центр обязателен) |
| `C5` | Хотя бы у одной стороны нет валидного пути/пары | Сосед не обязателен к commit; возможен базовый fallback |

## 5. Общий алгоритм пересчета (пошагово)

Алгоритм ниже описывает поведение **существующего** RetileEngine с учетом классификации ребер.

### Шаг 1. Формирование dirty set

При покраске одной или нескольких клеток RetileEngine вызывает `expandDirtySet()`:

- Для каждой покрашенной клетки (x, y) добавляются все клетки в квадрате [x-2..x+2, y-2..y+2] (Chebyshev R=2, до 25 клеток на покрашенную клетку).
- Если общий размер dirty set превышает 50% карты, переключается на full-pass (все клетки карты).
- В batch-режиме (fill, rectangle): dirty set = объединение R=2 расширений всех покрашенных клеток.

**Это не изменяется данной политикой.**

### Шаг 2. Обновление grid

Материалы покрашенных клеток обновляются в копии grid (immutability).

### Шаг 3. Пересчет каждой клетки в dirty set

Для **каждой** клетки в dirty set (без исключений) RetileEngine выполняет полный цикл:

1. Вычисление virtualBG для каждого кардинального ребра через `nextHop`.
2. Проверка доступности тайлсетов через `resolvePair` (direct-first, reverse fallback).
3. Выбор логического тайлсета: `selectTilesetForCell(fg, ownedEdges, registry)`.
4. Conflict resolution (S1/S2) при множественных BG.
5. Вычисление маски (base-mode или transition-mode).
6. Диагональный гейтинг.
7. Lookup фрейма через `getFrame`.
8. Frame-source resolution (renderTilesetKey).

### 5.1. Алгоритмические детали маски (8-bit -> blob-47)

Чтобы исключить неоднозначность, маска и фрейм считаются в строгом порядке:

1. Сначала строится `rawMask` из 8 направлений (внутренний порядок битов): `N=1, NE=2, E=4, SE=8, S=16, SW=32, W=64, NW=128`.
2. В base-mode (`bg == ''`) сначала применяется FG-сравнение: бит направления = 1, если сосед по этому направлению имеет тот же FG (для OOB принимается `bit=1`).
3. В engine owner-context для base-mode кардинальные foreign-ребра на non-owner стороне принудительно закрываются (`bit=1`).
4. В transition-mode (`bg != ''`) кардинали сначала подчиняются owner-side контракту:
   - если текущее ребро owned и его transitionType совпадает с выбранным BG, кардинальный бит открывается (`bit=0`);
   - иначе кардинальный бит принудительно закрывается (`bit=1`).
5. Диагонали проходят обязательный гейтинг:
   - `NW` может быть 1 только если `N=1` и `W=1`;
   - `NE` может быть 1 только если `N=1` и `E=1`;
   - `SE` может быть 1 только если `S=1` и `E=1`;
   - `SW` может быть 1 только если `S=1` и `W=1`.
6. После гейтинга получается `mask47Input`.
7. Для orientation `direct`: `frame = getFrame(mask47Input)`.
8. Для orientation `reverse`: `frame = getFrame((~mask47Input) & 0xFF)` (обратная диагональная/битовая ориентация).
   `getFrame()` выполняет `gateDiagonals()` внутри, поэтому после инверсии диагональный гейтинг применяется повторно на инвертированной маске.

Это соответствует инварианту из `design-015`: диагональ никогда не может "открыться сама по себе" без двух опорных кардиналей.

### Шаг 4. Запись результата

Сначала формируется полный набор пересчитанных кандидатов, затем применяется правило фиксации:

1. Центр фиксируется всегда.
2. Кардинальный сосед фиксируется только при классах C1/C2/C3.
3. Для C4/C5 кардинальный сосед может быть оставлен в предыдущем визуальном состоянии.
4. Диагональный сосед фиксируется только если обе прилегающие кардинали стабильны (`C1/C2/C3`):
   - NW фиксируется, если стабильны N и W.
   - NE фиксируется, если стабильны N и E.
   - SE фиксируется, если стабильны S и E.
   - SW фиксируется, если стабильны S и W.
5. Для остальных клеток dirty set (включая клетки на дистанции 2, которые не центр/кардиналь/диагональ) при paint-триггере фиксация не обязательна: если есть предыдущее visual/cache state, сохраняется предыдущее визуальное состояние.
6. Если у клетки нет предыдущего визуального состояния (первая инициализация кеша/слоя), она фиксируется даже вне обязательного commit-набора.
7. Для непокрасочных триггеров (например, full rebuild, T3/T4) фиксируются все клетки dirty set.
8. Спец-инвариант isolated-center-ring-solid применяется только для симметричной двухматериальной C2-пары (`A_B` и `B_A`, без моста): если центр зафиксирован как `frame=47`, все 8 клеток первого кольца вокруг центра фиксируются как `frame=1`.
9. Для мостовых C3-пар (пример: `deep_water <-> grass` через `water`) первый пояс не фиксируется принудительно в `frame=1`: кадры каждой клетки считаются по маске/коннекторам. Для центра frame получается из той же маски transition; например, для острова FG `1x1` результатом обычно будет `frame=47`, а для кластера -- connected-кадры.

Формально для paint-триггера:

- `centers = множество всех измененных клеток`.
- `mandatoryCommitSet = union_per_center( center ∪ stableCardinals(C1/C2/C3) ∪ stableDiagonals(по правилу угла) )`.
- `optionalCommitSet = dirtySet \ mandatoryCommitSet`.
- Для `optionalCommitSet` запись обязательна только при отсутствии предыдущего visual/cache state (инициализация).

После этого итог записывается в `layers[].frames` и `layers[].tilesetKeys` (immutable output), кеш обновляется только для committed-клеток. Для skipped-клеток из `optionalCommitSet` с предыдущим состоянием визуал и cache остаются прежними (без рассинхрона).

### Шаг 5. Возврат RetileResult

`RetileResult` содержит обновленные layers и patches для undo.

### 5.2. Почему пересчет локальный, но шире чем 1-радиус

В учебных примерах autotile обычно пересчитывают измененную клетку и ближайших соседей. В текущей архитектуре локальность сохранена, но зафиксирована как `Chebyshev R=2`:

1. Это все еще локальный пересчет, а не full-map rebuild.
2. Радиус 2 -- это консервативный инженерный контракт для единого поведения одиночной и batch-покраски, без ветвления логики по инструментам.
3. Для одиночной покраски R=2 не является строгой математической зависимостью маски клеток дистанции 1; это выбранный запас локальности для стабильной commit-фазы, детерминизма и уменьшения визуальных скачков в сложных локальных конфигурациях.
4. Поэтому R=2 трактуется как policy/engineering-контракт (single/batch consistency), а не как обязательная топологическая зависимость маски ближайших соседей.

## 6. Визуальное поведение по классам ребер

Классификация ребер используется как правило фиксации соседей после технического пересчета.

### 6.1. C1: Одноматериальный (A === B)

Оба соседа одного материала. Пересчет гарантированно дает корректный результат: маски и фреймы сшиваются.

**Ожидание**: Оба соседа входят в обязательный commit-набор (C1), но их `frame` может остаться прежним; швов не возникает.

### 6.2. C2: Симметричный прямой

Обе стороны имеют прямые тайлсеты. Это наиболее устойчивый переход.

**Ожидание**: Обе клетки на границе могут обновить тайлсет/маску/фрейм, но по каждому общему кардинальному ребру открытие выполняется только owner-side. Для isolated-center ring-solid используется правило из [§5, Шаг 4 (п.8)](#шаг-4-запись-результата).

**Пример**: `grass <-> water`. Существуют `grass_water` и `water_grass`.

### 6.3. C3: Устойчивый с мостом

Обе стороны валидны прямым способом через один и тот же промежуточный материал.

**Ожидание**: Обе клетки могут обновиться, owner-side контракт на каждом ребре остается единственным источником "открытия". Общий мост гарантирует совместимость transitionType.

Механика расчета transition для C3 через общий мост `M`:

1. Фрейм клетки определяется связностью собственной FG-области (same-FG кардинали).
2. Внешняя граница к материалу-партнеру учитывается как стык через мост `M`, а не как принудительный ring-solid.
3. `frame` вычисляется из этой маски без отдельного спец-`if` для `1x1`.
4. Для `1x1` и `2x2+` используются те же общие правила маски/коннекторов (без отдельных исключений для C3).

**Пример**: `deep_water <-> grass` через мост `water`. `nextHop(deep_water, grass) = water`, тайлсет `deep-water_water` (direct). `nextHop(grass, deep_water) = water`, тайлсет `grass_water` (direct).

### 6.4. C4: Смешанный валидный (reverse или разные мосты)

Обе стороны имеют resolved pair, но хотя бы одна сторона использует reverse orientation или мосты не совпадают.

**Ожидание**:

1. Центр обновляется всегда.
2. Для соседей изменение не фиксируется как обязательное.
3. Если движок пересчитал соседа и получил иной результат, это не считается целевым инвариантом.

Причины:

- Reverse orientation может давать визуально отличающийся угловой рисунок (инверсная маска).
- Разные мосты означают, что стороны опираются на разные transition-type ожидания.
- По каноническому edge-contract открытие должно происходить только на owner-side, поэтому двусторонняя "обязательная" перестройка для C4 не гарантируется.

**Пример**: `deep_water <-> water`. `nextHop(deep_water, water) = water`, тайлсет `deep-water_water` (direct). `nextHop(water, deep_water) = deep_water`, тайлсет `deep-water_water` (reverse, т.к. `water_deep-water` не существует как прямой тайлсет).

**Важно**: reverse-pair остается полноценным механизмом для вычисления перехода центра, но сам по себе не переводит ребро в устойчивый класс C2/C3.

### 6.5. C5: Частичный или невалидный

Хотя бы у одной стороны нет resolved pair (resolvePair вернул null).

**Ожидание**: Сторона без resolved pair не сможет отобразить переход. Она получит базовый тайлсет (no-transition mode, FG-equality mask). Визуально это выглядит как "жесткий обрыв" между материалами.

EdgeResolver вернет one-valid (если одна сторона рабочая) или null (если обе нерабочие). В обоих случаях класс ребра считается C5 с точки зрения политики фиксации соседей.

## 7. Взаимодействие с S1/S2 conflict resolution

### 7.1. S1 (Owner Reassign) работает на всех классах, но локально для клетки

S1 применяется внутри `CellTilesetSelector` только для текущей пересчитываемой клетки, когда у нее несколько конкурирующих BG. В текущей реализации S1:

- сокращает набор `ownedEdges` этой клетки по приоритетам;
- не записывает глобально новый owner-контракт ребра соседу;
- не создает отдельного обязательства commit для соседа сверх правил commit-фазы.

Следствие: S1 не переводит ребро C4/C5 в "обязательный commit" для соседа. Даже если центр локально изменил owner-side выбор, сосед фиксируется только по правилам commit-фазы (C1/C2/C3 для кардиналей, правило устойчивого угла для диагоналей, либо init fallback при отсутствии предыдущего состояния).

Это работает **независимо от класса ребра**:

- Для C2/C3: S1 локально выбирает устойчивого кандидата, при этом commit соседей обязателен по классу (без гарантии, что у каждого соседа сменится номер `frame`).
- Для C4: S1 локально снимает конфликт, но commit соседей остается необязательным по политике C4.
- Для C5: S1 может оставить единственный рабочий локальный вариант либо привести к fallback.

**Политика не ограничивает S1.** Conflict resolution и commit-policy — два разных уровня: S1 влияет на локальный выбор BG/тайлсета клетки, commit-policy определяет, какие из уже пересчитанных результатов фиксируются визуально.

### 7.2. S2 (BG Priority Fallback) не зависит от классификации

S2 выбирает BG с наивысшим приоритетом при неразрешенном конфликте. Это чисто детерминистический fallback, не учитывающий класс ребра.

## 8. Специальные условия (без сценариев)

### 8.1. Диагональные соседи

Диагональные клетки попадают в dirty set (R=2 включает их). Они пересчитываются, и для commit-фазы действует отдельное правило:

- Диагональные ребра не участвуют в EdgeResolver (только кардинальные N/E/S/W).
- Диагонали влияют на угловые биты маски через `gateDiagonals()`.
- Диагональ фиксируется только при "устойчивом угле": обе прилегающие к ней кардинали должны быть стабильными (`C1/C2/C3`).
- C2 isolated-center override для диагоналей определяется общим правилом [§5, Шаг 4 (п.8)](#шаг-4-запись-результата).
- Клетки на расстоянии 2 пересчитываются как часть консервативного R=2 контракта. Это не строгая зависимость маски distance=1, а выбор для единого и детерминированного поведения commit-фазы в single и batch сценариях.

См. примеры: [12.1](#example-12-1), [12.5](#example-12-5), [12.6](#example-12-6).

### 8.2. Выход за границы карты

Вне карты сосед не существует. `expandDirtySet` ограничивает координаты `[0, width-1]` и `[0, height-1]`. OOB-направления для масок: `bit = 1` (matching).

См. пример: [12.8](#example-12-8).

### 8.3. Перекраска в тот же материал

Если `newFG === oldFG`, RetileEngine все равно формирует dirty set и пересчитывает. Результат обычно идентичен предыдущему (детерминизм), но может отличаться, если окружение изменилось с момента последнего пересчета.

См. примеры: [12.1](#example-12-1), [12.2](#example-12-2) (повторная покраска тем же FG как no-op).

### 8.4. Цепочка покрасок

Каждая покраска -- независимое событие. Dirty set формируется заново. Клетка, которая не обновила визуал при предыдущей покраске (неустойчивый стык), может обновиться при следующей покраске, если попадет в новый dirty set и ее контекст изменится.

См. примеры: [12.5](#example-12-5) (расширение острова до `2x2+`), [12.6](#example-12-6) (смена контекста ребер).

### 8.5. Batch-режим (fill, rectangle)

Dirty set = объединение R=2 расширений всех покрашенных клеток. Все клетки пересчитываются. Порог 50% может сработать для крупных fill-операций, переключая на full-pass.
В batch-режиме `centers` = множество всех покрашенных клеток, а `mandatoryCommitSet` строится как объединение правил по каждому центру.

См. пример: [12.7](#example-12-7).

## 9. Правила детерминизма

1. **Порядок обработки**: RetileEngine обрабатывает dirty set в порядке flat index (top-to-bottom, left-to-right). S1 итерации используют тот же порядок.
2. **Direct-first**: При наличии обоих вариантов (direct и reverse) всегда выбирается direct (ADR Decision 7).
3. **Priority tie-break**: При равных приоритетах в EdgeResolver выбирается лексикографически меньший materialKey.
4. **Preference tie-break**: При равных BFS-путях Router использует preference array.
5. **Immutability**: Входной state никогда не мутируется. Результат -- новый объект.
6. **Полный вход детерминизма**: детерминизм определяется по полному входному состоянию: `grid + layers + cache + triggerType + options/materials/tilesets/priorities/preferences`.
7. **История-зависимость commit-policy**: для `optionalCommitSet` возможен retain предыдущего visual/cache state, поэтому одинаковый `grid` при разном предыдущем visual/cache state может дать разный результат (это ожидаемо по дизайну).
8. **Paint vs full-rebuild**: результат серии paint-операций может отличаться от full rebuild при одинаковом `grid`, потому что full rebuild коммитит все dirty-клетки, а paint применяет commit-фильтр. Это допустимое поведение политики.

Точная классификация каждой пары зависит от конкретных тайлсетов в наборе данных. Числа выше могут измениться если:

- Добавятся новые тайлсеты (reverse-пары станут direct, C4 перейдет в C2/C3).
- Изменится routing table (другие мосты, другая классификация).

Рекомендуется пересчитывать классификацию при каждом изменении набора тайлсетов. Для этого можно использовать утилиту:

```typescript
function classifyEdge(
  a: string, b: string,
  router: RoutingTable,
  registry: TilesetRegistry,
): 'C1' | 'C2' | 'C3' | 'C4' | 'C5' {
  if (a === b) return 'C1';

  const hopA = router.nextHop(a, b);
  const hopB = router.nextHop(b, a);

  const pairA = hopA !== null ? registry.resolvePair(a, hopA) : null;
  const pairB = hopB !== null ? registry.resolvePair(b, hopB) : null;

  if (pairA === null || pairB === null || hopA === null || hopB === null) return 'C5';

  const bothDirect = pairA.orientation === 'direct' && pairB.orientation === 'direct';

  if (!bothDirect) return 'C4';

  // Both direct. Check bridge conditions.
  if (hopA === b && hopB === a) return 'C2'; // Direct neighbors, no bridge

  const sameBridgeIntermediate =
    hopA === hopB &&
    hopA !== a &&
    hopA !== b;

  if (sameBridgeIntermediate) return 'C3'; // Same intermediate bridge M
  return 'C4'; // Different bridges or non-intermediate match
}
```

   - для отсутствующих соседей вне карты используется OOB-правило маски.
4. Что фиксируется:
   - обычные commit-правила из [§5](#5-общий-алгоритм-пересчета-пошагово) (центр, стабильные кардинали/диагонали);
   - dirty set обрезается по границам карты.
5. Ожидание:
   - нет выхода за пределы массива и нет артефактов от несуществующих соседей;
   - OOB-направления трактуются как matching (`bit=1`) при расчете маски.

## 13. Интеграция с RetileEngine

### 13.1. Точка интеграции

В текущей реализации политика фиксации соседей уже встроена в пайплайн `RetileEngine`:

1. `expandDirtySet()` формирует dirty set (Chebyshev R=2).
2. `rebuildCells()` сначала вычисляет кандидаты для всех клеток dirty set (phase 1), затем выполняет commit (phase 2).
3. На фазе commit применяется правило фиксации из [§5, Шаг 4](#шаг-4-запись-результата) без изменений.
4. Cache и visual state обновляются только для committed-клеток; skipped-клетки сохраняют предыдущий committed state.
5. S1/S2 влияют на локальный выбор кандидата клетки, но не расширяют mandatory commit-набор.

Это позволяет сохранить вычислительную корректность (полный пересчет в dirty set), но получить стабильный визуальный результат по правилам документа.

### 13.2. Будущая оптимизация (не для текущей реализации)

Если в будущем потребуется уменьшить dirty set для производительности, классификация может использоваться как **фильтр второго уровня**:

```
candidateDirtySet = expandDirtySet(changedCells)  // R=2 Chebyshev
filteredDirtySet = candidateDirtySet  // default: пересчитать все

// Опциональная оптимизация (не реализована):
// for each cell in candidateDirtySet:
//   if cell is NOT a painted cell:
//     if ALL edges to changed neighbors are C5:
//       skip cell (no visual change expected)
```

Такая оптимизация безопасна для C5 и требует отдельного профилирования для C4.
Для текущей версии обязательна только политика фиксации результата, без изменения геометрии dirty set.

## 14. Минимальный набор регрессионных проверок

Ниже -- тесты, которые валидируют корректность визуального поведения с учетом классификации ребер. Они **дополняют** (не заменяют) тесты из plan-015 Phase 8.

### 14.1. Проверки по классам

1. **C2**: Покраска `grass` в поле `water`. Ожидание: все 4 кардинальных соседа технически пересчитываются; при owner-side closure (prod priorities: `grass > water`) центр фиксируется как `grass_water` с `frame=47`; в этом C2-симметричном кейсе весь первый пояс вокруг центра (кардинали и диагонали) фиксируется как `frame=1`.
2. **C3**: Покраска `deep_sand` в поле `deep_water`. Ожидание: кардинальные соседи входят в mandatory commit через общий мост `water`; конкретный `frame` может совпасть с предыдущим, но шов остается устойчивым.
3. **C4**: Покраска `deep_water` в поле `water`. Ожидание: центр обновляется, соседи не обязаны менять визуал.
4. **C4**: Покраска `deep_sand` в поле `water`. Ожидание: центр обновляется, вода вокруг может остаться без изменений.
5. **Owner-side контракт**: для каждого кардинального ребра `A != B` проверяется, что ребро открыто только у owner-клетки; у не-owner стороны это ребро закрыто.
6. **Диагональный гейтинг**: кейсы с одиночной диагональю (`NE=1` при `N=0` или `E=0`) не должны сохранять диагональный бит после `gateDiagonals`.
7. **Reverse-ориентация**: при отсутствии direct-пары и наличии reverse-пары фрейм берется через `getFrame((~mask47Input) & 0xFF)` после `gateDiagonals(rawMask)`, а не через прямой lookup.
8. **Детерминизм**: повторный запуск тех же операций дает идентичный результат (тайлсеты, фреймы, маски).
9. **Batch**: Fill 100 клеток `grass` в поле `water`. Все клетки в объединенном R=2 dirty set пересчитаны.
10. **Диагональный commit-угол**: единичная покраска центра в окружении, где стабильны только N и E. Ожидание: commit обязателен для NE-диагонали, но не обязателен для NW/SE/SW.
11. **C3 bridge island**: `water` поле + кольцо `grass` + центр `deep_water`. Ожидание: первый пояс и диагонали считаются по коннекторам (не принудительно `1`), а в частном случае острова `1x1` transition-маска дает центру `frame=47`.

### 14.2. Регрессионные проверки (из design-015 AC9)

12. **P1**: deep-water fill + soil rectangle -- корректные deep-water_water переходы.
13. **P2**: soil square + water lake -- корректные water_grass blob-47 corners.
14. **P3**: deep-water | soil | sand strips -- маршрутизация через мосты.
