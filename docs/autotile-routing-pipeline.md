# Autotile Routing Pipeline — как автоматически подбирать тайлы переходов на карте

## Проблема

Представьте 2D-редактор карт: пользователь рисует материалами — трава, вода, песок, камень. На границах между ними нужны **плавные переходы** — не просто резкий стык «трава | вода», а красивые плавные края, углы, коридоры.

Для этого используется **autotiling** — система, которая автоматически подбирает нужный кадр из спрайтшита в зависимости от окружения клетки.

Но классический autotiling работает только для **одного** типа перехода (есть материал / нет материала). А что если у нас 10+ материалов и не для каждой пары нарисован спрайтшит перехода? Как визуально связать песок с водой, если есть только `песок→грязь` и `грязь→вода`?

---

## Решение: маршрутизация материалов

Идея: относиться к материалам как к **узлам в графе**, а к тайлсетам переходов — как к **рёбрам**. Тогда задача «как визуально связать два материала» превращается в задачу **поиска кратчайшего пути** в графе.

---

## Пайплайн из 5 шагов

### Шаг 1 — Реестр тайлсетов

**Что это.** Централизованный иммутабельный индекс всех доступных тайлсетов. Каждый тайлсет объявляет пару материалов, для которых он может отрисовать переход: foreground (FG) поверх background (BG). Например:

- `grass_dirt` — трава на фоне грязи (тайлсет перехода)
- `dirt_water` — грязь на фоне воды (тайлсет перехода)
- `grass` (standalone) — трава без переходов, полная заливка (базовый тайлсет)

Если тайлсет `A_B` не существует, но существует `B_A`, он используется в **reverse-ориентации** (зеркально). Реестр обрабатывает это внутренне: на вопрос «можно ли отрисовать траву на воде?» он проверяет и `grass_water` (прямой), и `water_grass` (обратный), и возвращает первый найденный с флагом ориентации.

**Зачем нужен.** Все последующие шаги задают вопросы вроде «есть ли тайлсет для этой пары?», «какой базовый тайлсет у материала?», «покажи все известные пары переходов». Реестр отвечает на каждый за O(1) благодаря заранее построенным хеш-таблицам, поэтому остальному пайплайну никогда не нужно сканировать сырой массив тайлсетов.

### Шаг 2 — Построение графа совместимости

**Что это.** **Ненаправленный граф**, построенный из реестра. Узлы — материалы. Ребро A↔B существует, если в реестре зарегистрирован тайлсет для пары A_B или B_A (в любой ориентации). Параллельно строится **направленный граф**: ребро A→B существует только при наличии прямого тайлсета A_B.

**Зачем нужен.** Граф — это вход для алгоритма маршрутизации в следующем шаге. Без него невозможно обнаружить multi-hop связи между материалами. Ненаправленный граф отвечает на вопрос «какие материалы вообще могут визуально переходить друг в друга — напрямую или через посредников». Направленный граф используется позже для решений рендеринга — знание, прямой тайлсет или обратный, влияет на правильную ориентацию спрайта.

**Пример.** Даны тайлсеты `grass_dirt`, `dirt_water` и `sand_dirt`. Граф совместимости содержит рёбра: grass↔dirt, dirt↔water, sand↔dirt. Прямого ребра grass↔water или sand↔water нет, но пути через dirt существуют.

### Шаг 3 — BFS-маршрутизация (all-pairs)

**Что это.** Классический **Breadth-First Search из каждого узла** по графу совместимости — алгоритм all-pairs shortest path. Для каждой пары материалов (A, B) результат сохраняется в таблице маршрутизации как `nextHop(A, B)` — первый материал на кратчайшем пути от A к B.

> `nextHop(sand, water) → dirt`

Это значит: «чтобы визуально соединить песок с водой, первый шаг — грязь, поэтому на границе используй тайлсет `sand_dirt`».

При нескольких кратчайших путях одинаковой длины — tie-breaking по настраиваемому **массиву предпочтений** (например, предпочитать грязь камню в качестве посредника). Это гарантирует детерминированный, управляемый художниками результат.

**Зачем нужен.** Это ключевая идея всего пайплайна. Классический autotiling требует выделенный спрайтшит для каждой пары соседних материалов. При M материалах это до M×(M−1)/2 спрайтшитов — комбинаторный взрыв. Таблица маршрутизации устраняет это: художникам нужно рисовать тайлсеты только для **прямых соседей** в графе, а система автоматически выстраивает multi-hop переходы для далёких материалов.

**Почему именно BFS?** BFS находит кратчайший путь в невзвешенном графе, что напрямую соответствует нашей цели: минимизировать количество промежуточных материалов в цепочке визуального перехода. Он работает за O(V + E) на один источник, O(V × (V + E)) суммарно — достаточно быстро для типичных 10–30 материалов в игре.

### Шаг 4 — Разрешение владельца границы (Edge Resolution)

**Что это.** Для каждой **кардинальной границы** (N/E/S/W) между двумя клетками с разными материалами алгоритм определяет, какая клетка «владеет» переходом — т.е. какая клетка отвечает за отрисовку граничного спрайта.

Алгоритм разрешения:

1. **Вычислить виртуальные фоны.** Обе стороны запрашивают таблицу маршрутизации: клетка A вычисляет `nextHop(A_материал, B_материал)`, клетка B вычисляет `nextHop(B_материал, A_материал)`.
2. **Проверить доступность тайлсета.** Кандидат валиден только если `registry.resolvePair(материал, virtualBG)` вернул результат.
3. **Разрешить конфликт.** Если только у одной стороны есть валидный тайлсет — она побеждает по умолчанию. Если обе стороны валидны — побеждает материал с **бо́льшим приоритетом** (приоритет настраивается художником). При равных приоритетах — **лексикографический порядок** по ключам материалов как детерминированный тай-брейкер, чтобы результат был идентичен независимо от того, какая клетка запрашивает.

**Зачем нужен.** Без определения владельца обе клетки пытались бы рисовать одну и ту же границу — получая двойной рендеринг, z-fighting или визуальные швы. Ровно одна клетка должна отвечать за каждую границу. Владение также используется при вычислении маски Blob-47 в следующем шаге: биты маски клетки корректируются в зависимости от того, какими границами она владеет, предотвращая некорректные угловые паттерны на переходных границах.

### Шаг 5 — Blob-47 Autotiling

Теперь нужно превратить решения маршрутизации в конкретный кадр спрайта. Здесь вступает **Blob-47** — техника автотайлинга для квадратных сеток, которая определяет, какой спрайт рисовать, на основе **того, какие из 8 соседей клетки имеют тот же террейн**. Называется «47», потому что после применения правил diagonal gating остаётся ровно **47 уникальных конфигураций**.

#### 8-битная маска соседей

Каждая клетка проверяет все 8 соседей и строит **8-битную маску** с такими весами:

```
 NW(128)  N(1)   NE(2)
  W(64)  [cell]   E(4)
 SW(32)   S(16)  SE(8)
```

Если сосед имеет тот же террейн — его бит равен 1, иначе — 0.

Например, если у клетки травы соседи-трава на Севере, Востоке и Северо-востоке, сырая маска:

```
N(1) + NE(2) + E(4) = 7
```

**Почему битовая маска?** Она кодирует все 256 возможных комбинаций 8 соседей в одно целое число, позволяя O(1) lookup в предвычисленной таблице кадров. Без условий, без ветвлений — просто индекс в таблице.

#### Diagonal Gating

Ключевая идея: **диагональный сосед учитывается только если оба смежных кардинала тоже присутствуют**. Без этого правила появляются визуальные артефакты — угловой тайл рисуется там, где реального угла нет.

Представьте клетку, у которой трава на Севере и Северо-востоке, но не на Востоке. Рисовать NE-угол здесь неправильно — нет реального замкнутого угла, есть только два отдельных края. Diagonal gating предотвращает это, сбрасывая бит NE при отсутствии N или E.

Правила гейтинга:
- **NE** считается только если установлены **N** и **E**
- **SE** считается только если установлены **S** и **E**
- **SW** считается только если установлены **S** и **W**
- **NW** считается только если установлены **N** и **W**

```
gated_mask = оставить только кардинальные биты (N, E, S, W)
если N и E установлены и NE был установлен → добавить NE
если S и E установлены и SE был установлен → добавить SE
если S и W установлены и SW был установлен → добавить SW
если N и W установлены и NW был установлен → добавить NW
```

Это сокращает 256 возможных масок до ровно **47 валидных конфигураций**.

**Почему именно 47?** 4 кардинальных бита (2⁴ = 16 кардинальных комбинаций). Для каждой кардинальной комбинации могут варьироваться только те диагонали, которые примыкают к двум существующим кардиналам. Математика даёт ровно 47 валидных гейтированных масок — любая другая невозможна при гейтинге.

#### 47 конфигураций

47 валидных масок отображаются на конкретные формы тайлов:

| Диапазон кадров | Форма | Описание |
|-----------------|-------|----------|
| 1 | Сплошной | Все 8 соседей присутствуют |
| 2–16 | Центральные варианты | Все 4 кардинала на месте, разные комбинации углов |
| 17–20 | T-перекрёсток, открыт W | N+E+S есть, W нет, варианты углов |
| 21–24 | T-перекрёсток, открыт N | E+S+W есть, N нет, варианты углов |
| 25–28 | T-перекрёсток, открыт E | N+S+W есть, E нет, варианты углов |
| 29–32 | T-перекрёсток, открыт S | N+E+W есть, S нет, варианты углов |
| 33–34 | Коридоры | Вертикальный (N+S) или горизонтальный (E+W) |
| 35–42 | L-углы | Два смежных кардинала, с заполненным углом / без |
| 43–46 | Тупики | Только один кардинальный сосед |
| 47 | Изолированный | Соседей нет |

#### Lookup-таблица кадров

Вместо вычисления кадра в рантайме через условия, при загрузке строится **lookup-таблица из 256 записей**:

```
FRAME_TABLE[gated_mask] → индекс кадра (0–47)
```

Кадр 0 зарезервирован для «пустого» (нет террейна). Кадры 1–47 соответствуют вариантам автотайла. Невалидные маски отображаются на изолированный кадр (47) как fallback.

Полное вычисление для одной клетки:

```
raw_mask = просканировать 8 соседей, установить биты для совпадающих
gated_mask = применить diagonal gating
frame = FRAME_TABLE[gated_mask]
```

Это **O(1)** на клетку — построение битовой маски + один lookup в массиве.

**Почему lookup-таблица, а не цепочка if/else?** При 47 уникальных случаях ручное кодирование условий было бы подвержено ошибкам и медленно. Таблица на 256 байт тривиально предвычисляется, помещается в L1-кэш и сводит определение кадра к одному обращению в память.

#### Раскладка спрайтшита

Спрайтшит каждого террейна — сетка **12 колонок × 4 строки = 48 кадров**:

```
Кадр 0  = пустой/прозрачный
Кадр 1  = сплошной (все соседи)
Кадр 2  = нет NW-угла
...
Кадр 47 = изолированный (нет соседей)
```

Порядок идёт от **максимально связанного** (сплошной, кадр 1) к **минимально связанному** (изолированный, кадр 47). Эта конвенция единая для всех типов террейна, поэтому художники всегда знают, какой кадр соответствует какой форме.

#### Обработка границ карты

Клетки у края карты имеют меньше 8 соседей. Две стратегии:
- **За границей = совпадение** (по умолчанию): края карты выглядят бесшовно, как будто террейн продолжается за пределы
- **За границей = несовпадение**: края карты показывают явные границы (стиль острова)

Выбор — один булевый флаг при вычислении маски, никакой дополнительной логики не нужно.

#### Почему не Blob-255?

Blob-255 использует все 256 масок без diagonal gating, требуя 256 спрайтов на террейн. Blob-47 достигает практически идентичного визуального качества с **6× меньшим количеством спрайтов** — существенная экономия на производстве арта. Diagonal gating убирает конфигурации, которые визуально были бы неотличимы или некорректны.

#### Расширение для режима переходов

Всё описанное выше работает для одного типа террейна («мой материал или нет»). Routing pipeline расширяет это для **мультиматериальных переходов**:

- **Base mode** (нет перехода): маска проверяет, совпадает ли сосед с FG-материалом
- **Transition mode** (выбран BG через маршрутизацию): маска строится относительно **виртуального фона**. Дополнительно **ownership gating** корректирует кардинальные биты в зависимости от того, кто владеет границей (из Шага 4)

Это значит, что одна и та же таблица кадров Blob-47 переиспользуется и для обычных заливок, и для сложных маршрутизированных переходов — меняется только способ вычисления маски.

**Почему это важно.** Переиспользование одного и того же спрайтшита на 47 кадров и для базового террейна, и для маршрутизированных переходов означает нулевые дополнительные арт-ресурсы для мультиматериальной поддержки. Вся сложность мультиматериальности обрабатывается чисто в коде на уровне маски.

## Оптимизация перерисовки

При изменении одной клетки не нужно пересчитывать всю карту. Система использует **умную грязную область**:

**1. Chebyshev R=2 расширение.** При изменении клетки «грязной» помечается квадрат 5×5 вокруг неё. Почему R=2, а не R=1? Потому что diagonal gating означает, что кадр клетки зависит от соседей её соседей. Если клетка X изменилась, маска её соседа Y изменится, а диагональный сосед Z может тоже быть затронут через кардинальную связь Y с Z. Два хопа в расстоянии Чебышёва покрывают полный радиус влияния.

**2. Классификация стабильности границ.** Не все грязные клетки реально нуждаются в перерисовке. Каждой кардинальной границе присваивается класс стабильности:
   - **C1** — одинаковый материал (всегда стабильно — нечего перерисовывать)
   - **C2** — симметричная прямая пара; обе стороны имеют прямые тайлсеты (стабильно)
   - **C3** — пара через общий промежуточный хоп; обе стороны прямые (стабильно)
   - **C4** — обе стороны resolved, но маршрутизация может измениться при редактировании соседа (нестабильно)
   - **C5** — хотя бы у одной стороны нет маршрута (невалидно)

**Зачем классифицировать?** Класс стабильности определяет **политику selective commit**: коммитятся только клетки рядом со стабильными границами (C1/C2/C3). C4/C5 соседи не трогаются — их закэшированное визуальное состояние сохраняется. Это предотвращает артефакты «ряби», когда редактирование одной клетки вызывает каскад визуальных изменений по карте через нестабильные цепочки маршрутизации.

**3. Selective commit.** Из dirty set реально перерисовываются только:
   - Сама изменённая клетка (всегда)
   - Кардинальные соседи с C1/C2/C3 границами
   - Диагональные соседи, но только если оба смежных кардинала стабильны
   - C4/C5 соседи **не трогаются** — их визуальное состояние сохраняется из кэша

**4. Per-cell кэш.** Для каждой клетки хранится последний результат вычислений: ключ выбранного тайлсета, ключ рендер-тайлсета, индекс кадра, виртуальный фон и ориентация. Reverse-индексы (ключ тайлсета → множество клеток) позволяют мгновенно находить все клетки при обновлении или удалении тайлсета — без полного сканирования карты.

## Итого

| Компонент | Сложность | Назначение |
|-----------|-----------|------------|
| Реестр | O(T) | Индексация T тайлсетов в хеш-таблицы |
| Граф | O(T) | Построение смежности из T пар |
| BFS-роутер | O(M² · E) | All-pairs кратчайший путь для M материалов |
| Edge Resolution | O(1) per edge | Определение владельца границы по кардиналам |
| Blob-47 | O(1) per cell | Lookup-таблица `mask → frame` |
| Dirty propagation | O(changed × 25) | Chebyshev 5×5 на каждую изменённую клетку |

Весь пайплайн нужен для одного: **пользователь просто рисует кисточкой, а система в реальном времени подбирает правильные тайлы переходов даже для материалов, между которыми нет прямого тайлсета** — через маршрутизацию по промежуточным материалам.
