# ADR-0014: AI-диалоги NPC через OpenAI + Vercel AI SDK

## Статус

Proposed

## Контекст

В рамках Design-020 реализована базовая диалоговая система с псевдо-стримингом (echo-режим): сервер повторяет сообщения игрока по словам без участия AI. Следующий шаг -- замена echo-обработчика на реальные AI-ответы, добавление персистентности диалогов и системы персон NPC.

Ключевые архитектурные вопросы:
1. Какого AI-провайдера и SDK использовать для генерации ответов NPC?
2. Как интегрировать AI-стриминг с существующим Colyseus WebSocket-протоколом?
3. Как хранить историю диалогов для памяти NPC?
4. Как описывать персону NPC для формирования системного промпта?

GDD проекта предполагает использование Anthropic Claude (Haiku для рутины, Sonnet для диалогов), однако на этапе MVP выбор AI-провайдера не является необратимым благодаря абстракции AI SDK.

---

## Решение 1: Выбор AI-провайдера и SDK

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Использовать OpenAI GPT-4o-mini через Vercel AI SDK (`streamText()` из пакета `ai`) |
| **Почему сейчас** | Echo-обработчик -- временная заглушка; реальные AI-ответы -- следующее требование по roadmap |
| **Почему это** | AI SDK абстрагирует провайдера через единый интерфейс `streamText()` -- переключение на Claude требует только замены импорта модели. GPT-4o-mini экономичен для MVP (~$0.15/1M input tokens) |
| **Известные неизвестные** | Качество RP-диалогов GPT-4o-mini для NPC в контексте фермерской RPG (может потребоваться fine-tuning промптов) |
| **Kill criteria** | Если стоимость API превышает $0.50/час при 10 одновременных игроках, или качество ответов неприемлемо после 3 итераций промпт-инжиниринга |

### Рассмотренные варианты

#### Вариант A: Прямой вызов OpenAI REST API

- **Описание**: Использовать `openai` npm-пакет напрямую для вызова Chat Completions API.
- **Плюсы**:
  - Минимум зависимостей (один пакет `openai`)
  - Полный контроль над HTTP-запросами и парсингом SSE
  - Документация OpenAI SDK хорошо описывает streaming
- **Минусы**:
  - Привязка к OpenAI -- смена провайдера требует переписывания клиента
  - Ручная обработка стриминга (SSE-парсинг, буферизация чанков)
  - Нет абстракции для будущего переключения на Claude (требование GDD)
  - Дублирование логики retry, timeout, error handling

#### Вариант B (Выбран): Vercel AI SDK с OpenAI провайдером

- **Описание**: Использовать `streamText()` из `ai` пакета с `@ai-sdk/openai` провайдером. Единый интерфейс для любого LLM-провайдера.
- **Плюсы**:
  - Провайдер-агностичный API: `streamText({ model: openai('gpt-4o-mini'), ... })` заменяется на `streamText({ model: anthropic('claude-3-haiku'), ... })` одной строкой
  - Встроенная обработка стриминга через `textStream` (AsyncIterable)
  - Встроенный `AbortSignal` для отмены
  - `onAbort` колбек для cleanup
  - Активно поддерживается Vercel, 5000+ code snippets в документации
  - Уже используется на клиенте (AI SDK `useChat` в ChatModal из Design-020)
- **Минусы**:
  - Дополнительная зависимость (`ai` + `@ai-sdk/openai`)
  - AI SDK v5 изменил API (Responses API по умолчанию вместо Chat Completions) -- нужно использовать `openai.chat()` если требуется обратная совместимость
  - Абстракция может скрывать детали провайдер-специфичных ошибок

#### Вариант C: Anthropic Claude напрямую (как в GDD)

- **Описание**: Использовать `@anthropic-ai/sdk` для вызова Claude API напрямую.
- **Плюсы**:
  - Соответствие GDD (Claude Haiku для рутины, Sonnet для диалогов)
  - Claude лучше справляется с roleplay-сценариями согласно бенчмаркам
  - Anthropic SDK хорошо типизирован
- **Минусы**:
  - Claude Haiku дороже GPT-4o-mini для MVP (~$0.25 vs ~$0.15 за 1M input tokens)
  - Привязка к одному провайдеру
  - GDD можно будет следовать позже -- AI SDK позволяет переключиться в одну строку
  - Требует отдельного API-ключа Anthropic (OpenAI API-ключ доступнее для разработчиков)

### Обоснование

Вариант B выбран как оптимальный компромисс между стоимостью MVP, гибкостью и совместимостью с GDD. AI SDK обеспечивает провайдер-агностичный интерфейс: переключение на Claude (требование GDD) -- замена одного импорта. GPT-4o-mini оптимален по соотношению цена/качество для MVP-диалогов. Клиентская часть уже использует AI SDK (`useChat` в ChatModal), что обеспечивает единообразие стека.

### Последствия

**Положительные**:
- Единый стек AI SDK на клиенте и сервере
- Переключение провайдера без изменения бизнес-логики
- Экономичный MVP (GPT-4o-mini)

**Отрицательные**:
- Временное отклонение от GDD (OpenAI вместо Claude)
- Зависимость от Vercel AI SDK (абстракция)

**Нейтральные**:
- Два новых npm-пакета в `apps/server` (`ai`, `@ai-sdk/openai`)

## Решение 2: Паттерн интеграции AI-стриминга с Colyseus

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Серверный `streamText()` стримит через существующий Colyseus WebSocket-протокол (DIALOGUE_STREAM_CHUNK сообщения) |
| **Почему сейчас** | Необходимо заменить setTimeout-based echo на реальный AI-стриминг, сохраняя существующий протокол |
| **Почему это** | Zero client changes -- ColyseusTransport на клиенте backend-agnostичен. Протокол DIALOGUE_STREAM_CHUNK/DIALOGUE_END_TURN остаётся неизменным |
| **Известные неизвестные** | Оптимальный размер чанков (по слову, по предложению, по токену) для плавного UX |
| **Kill criteria** | Если задержка между чанками > 500ms создаёт ощущение "зависания" UI |

### Рассмотренные варианты

#### Вариант A: HTTP Server-Sent Events параллельно с Colyseus

- **Описание**: Добавить Express/Hono HTTP endpoint для AI-стриминга; клиент подключается к двум каналам.
- **Плюсы**:
  - Стандартный SSE-паттерн для AI-стриминга
  - Независимость AI-канала от game state
- **Минусы**:
  - Дублирование транспорта (WebSocket + HTTP SSE)
  - Клиент должен координировать два соединения
  - Нарушение принципа "один WebSocket для всего game state"
  - Потребует значительных изменений в ColyseusTransport (Design-020)

#### Вариант B (Выбран): Стриминг через Colyseus WebSocket

- **Описание**: `streamText()` на сервере возвращает `textStream` (AsyncIterable). `for await (const chunk of textStream)` отправляет каждый чанк как `DIALOGUE_STREAM_CHUNK` через `client.send()`. После завершения -- `DIALOGUE_END_TURN`.
- **Плюсы**:
  - Zero client changes: существующий ColyseusTransport и ChatModal работают без модификаций
  - Единый транспорт для всего (game state + dialogue)
  - AbortController для отмены при disconnect/close
  - Простая замена echo-обработчика: вместо setTimeout-цепочки -- async for loop
- **Минусы**:
  - Colyseus message overhead на каждый чанк (header ~20 bytes)
  - Нет backpressure: если клиент медленный, чанки буферизуются в Colyseus

#### Вариант C: Буферизация полного ответа + единоразовая отправка

- **Описание**: Дождаться полного AI-ответа, затем отправить одним сообщением.
- **Плюсы**:
  - Простота реализации
  - Нет проблем с частичными стримами
- **Минусы**:
  - Потеря UX стриминга (пользователь ждёт 2-5 секунд без обратной связи)
  - Нарушает установленный контракт (DIALOGUE_STREAM_CHUNK для word-by-word display)
  - Не соответствует паттерну ChatModal с `useChat` (ожидает streaming)

### Обоснование

Вариант B -- естественная эволюция echo-обработчика из Design-020. `setTimeout` заменяется на `for await`, протокол остаётся идентичным. Клиент не знает, что вместо echo теперь AI -- архитектурная инверсия зависимостей в действии.

## Решение 3: Персистентность диалогов

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Постоянное хранение в PostgreSQL через Drizzle ORM: таблицы `dialogue_sessions` и `dialogue_messages` |
| **Почему сейчас** | AI-диалоги без истории бессмысленны: NPC не помнит предыдущих разговоров. GDD требует memory stream для NPC |
| **Почему это** | PostgreSQL + Drizzle -- существующий стек проекта. Две таблицы покрывают текущие и будущие потребности (memory retrieval, reflection) |
| **Известные неизвестные** | Оптимальный размер контекстного окна для загрузки истории (10/20/50 последних сообщений) |
| **Kill criteria** | Если время загрузки истории > 100ms при 1000+ сообщениях на пару bot-player |

### Рассмотренные варианты

#### Вариант A: Без персистентности (in-memory only)

- **Описание**: Хранить историю диалогов только в памяти ChunkRoom. При перезапуске сервера история теряется.
- **Плюсы**:
  - Нулевая латентность -- нет DB-запросов
  - Простейшая реализация
  - Нет миграций
- **Минусы**:
  - Потеря всей истории при рестарте сервера
  - Невозможно реализовать NPC-память (требование GDD)
  - Нет данных для будущей системы reflection
  - Масштабирование на несколько серверов невозможно

#### Вариант B: Redis с TTL

- **Описание**: Хранить историю в Redis как JSON-списки с TTL 7 дней.
- **Плюсы**:
  - Быстрое чтение/запись
  - TTL автоматически очищает старые данные
  - Разделение нагрузки между PostgreSQL и Redis
- **Минусы**:
  - Redis не используется в текущем стеке (нужно добавить инфраструктуру)
  - JSON в Redis -- нет реляционных запросов, нет JOIN с npc_bots/users
  - TTL противоречит GDD: NPC-память должна быть долгосрочной
  - Дополнительная зависимость для MVP

#### Вариант C (Выбран): PostgreSQL через Drizzle ORM

- **Описание**: Две новые таблицы в `packages/db`: `dialogue_sessions` (сессии диалогов) и `dialogue_messages` (сообщения). FK на `npc_bots.id` и `users.id`.
- **Плюсы**:
  - Соответствие существующему стеку (PostgreSQL + Drizzle)
  - Реляционные связи: bot → sessions → messages
  - Фундамент для NPC memory stream (GDD): `SELECT messages WHERE importance > threshold ORDER BY recency`
  - Индексируемость по `botId`, `createdAt` для эффективного retrieval
  - Drizzle schema -- единый source of truth для TypeScript типов
- **Минусы**:
  - DB-запросы добавляют латентность (~5-20ms)
  - Дополнительная миграция
  - Рост объёма данных со временем (потребуется архивация/purge в будущем)

### Обоснование

Вариант C следует принципу проекта "одна сущность -- одна таблица" (паттерн `users`, `player_positions`, `maps`, `npc_bots`). Две таблицы (`dialogue_sessions` + `dialogue_messages`) -- минимальная нормализованная схема для хранения диалогов. Связь с `npc_bots` через FK обеспечивает целостность. Латентность DB-запросов компенсируется тем, что запись происходит fire-and-forget (не блокирует стриминг), а чтение истории -- один запрос при начале диалога.

## Решение 4: Система персон NPC

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Расширить таблицу `npc_bots` колонками `personality`, `role`, `speechStyle` для формирования системного промпта |
| **Почему сейчас** | AI без персоны генерирует generic-ответы. Персона -- минимально необходимый контекст для RP-диалогов |
| **Почему это** | Расширение существующей таблицы (а не новая) -- минимальное изменение. Три колонки покрывают основные аспекты персоны по GDD (seed persona) |
| **Известные неизвестные** | Достаточно ли текстового описания или потребуется структурированный JSON для seed persona (как в GDD) |
| **Kill criteria** | Если текстовые описания создают слишком разнообразные/несогласованные ответы и потребуется структурированный формат |

### Рассмотренные варианты

#### Вариант A: Отдельная таблица `npc_personas`

- **Описание**: Создать отдельную таблицу `npc_personas` с FK на `npc_bots.id`.
- **Плюсы**:
  - Чистое разделение: бот (позиция, спавн) vs. персона (характер, стиль)
  - Переиспользование персон для нескольких ботов
  - Совместимо с GDD-концепцией "seed persona"
- **Минусы**:
  - Оверинжиниринг для MVP: один бот = одна персона (нет переиспользования)
  - JOIN при каждом начале диалога
  - Дополнительная таблица и миграция

#### Вариант B: JSON-поле `persona` в `npc_bots`

- **Описание**: Добавить одну JSONB-колонку `persona` в `npc_bots` со структурой `{ personality: string, role: string, speechStyle: string }`.
- **Плюсы**:
  - Гибкость: можно добавлять поля без миграций
  - Одна колонка вместо трёх
- **Минусы**:
  - JSONB не типизирован на уровне DB (runtime-валидация)
  - Drizzle не генерирует TypeScript типы для JSONB-содержимого автоматически
  - Нарушает паттерн проекта: другие таблицы используют явные колонки, не JSON

#### Вариант C (Выбран): Три текстовые колонки в `npc_bots`

- **Описание**: Добавить `personality` (text, nullable), `role` (varchar(64), nullable), `speechStyle` (text, nullable) в существующую таблицу `npc_bots`.
- **Плюсы**:
  - Соответствие паттерну проекта (явные типизированные колонки)
  - Drizzle автоматически генерирует TypeScript типы
  - Nullable -- обратная совместимость (существующие боты продолжают работать)
  - Нет JOIN -- все данные бота в одном SELECT
- **Минусы**:
  - Миграция ALTER TABLE
  - Три колонки вместо одной -- но каждая имеет чёткую семантику
  - При эволюции в полноценную seed persona (GDD) может потребоваться рефакторинг

### Обоснование

Вариант C следует паттерну проекта (явные колонки с типизацией). Nullable-колонки обеспечивают обратную совместимость: существующие боты без персоны продолжают работать (AI получает default system prompt). При переходе к полноценной NPC-системе из GDD колонки можно мигрировать в seed_persona JSON или отдельную таблицу npc_personas.

## Сравнительная матрица (все решения)

| Решение | Выбранный вариант | Главное преимущество | Главный компромисс |
|---------|-------------------|---------------------|-------------------|
| #1 AI-провайдер | AI SDK + OpenAI GPT-4o-mini | Провайдер-агностичность, экономичность | Временное отклонение от GDD (OpenAI vs Claude) |
| #2 Интеграция стриминга | Через Colyseus WebSocket | Zero client changes | Message overhead на каждый чанк |
| #3 Персистентность | PostgreSQL (sessions + messages) | Реляционная модель, фундамент для NPC memory | DB-латентность (~5-20ms) |
| #4 Персона NPC | Три колонки в `npc_bots` | Соответствие паттерну проекта, типизация | Миграция ALTER TABLE |

## Руководство по имплементации

- Использовать `streamText()` из `ai` пакета -- единый entry point для AI-генерации
- Использовать `openai.chat('gpt-4o-mini')` из `@ai-sdk/openai` для явного указания Chat Completions API
- Для переключения провайдера достаточно заменить `openai.chat('gpt-4o-mini')` на `anthropic('claude-3-haiku-20240307')` -- никаких изменений в бизнес-логике
- DialogueService получает зависимости через конструктор (DI): модель, конфигурация
- DB-операции (сохранение сообщений) выполняются fire-and-forget -- не блокируют стриминг ответа
- Использовать `AbortController` для отмены AI-запроса при disconnect/close (вместо `setTimeout` cleanup)
- Системный промпт формируется из bot persona + recent history + game context -- чистое разделение ответственностей
