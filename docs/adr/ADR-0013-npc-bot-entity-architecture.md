# ADR-0013: Архитектура сущности NPC бот-компаньона

## Статус

Proposed

## Контекст

В рамках PRD-009 (NPC Bot-Companion System) необходимо добавить серверно-управляемых бот-компаньонов на хоумстеды игроков. Это первая NPC-сущность в проекте и прекурсор полноценной NPC-системы, описанной в `docs/documentation/design/systems/npc-service.md`.

Ключевые архитектурные вопросы:
1. Как представить ботов в Colyseus-состоянии комнаты?
2. Где разместить серверную логику ботов?
3. Как хранить ботов в базе данных?
4. Какой алгоритм перемещения использовать?
5. Как рендерить ботов на клиенте?
6. Какой протокол взаимодействия применить?

Текущая архитектура: ChunkRoom управляет MapSchema `players`, World хранит авторитативное состояние, PlayerManager на клиенте создаёт PlayerSprite для удалённых игроков.

---

## Решение 1: Представление ботов в Colyseus-состоянии

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Использовать отдельную `bots` MapSchema в ChunkRoomState, параллельную `players` |
| **Почему сейчас** | Бот-компаньоны — первая NPC-сущность; заложить правильное разделение доменов сейчас дешевле, чем рефакторить позже |
| **Почему это** | Чистое разделение игроков и NPC предотвращает ошибки маршрутизации сообщений и упрощает будущее расширение |
| **Известные неизвестные** | Влияние второй MapSchema на размер Colyseus-патча при большом количестве ботов |
| **Kill criteria** | Если Colyseus-патч-рейт деградирует > 20% при 10 ботах в комнате |

### Рассмотренные варианты

#### Вариант A: Смешивание в `players` MapSchema

- **Описание**: Добавить ботов в существующую MapSchema `players` с дополнительным полем `isBot: boolean`.
- **Плюсы**:
  - Минимальные изменения в клиентском коде (PlayerManager обрабатывает единообразно)
  - Физическое столкновение бота и игрока работает автоматически через существующую систему
  - Один цикл обработки для всех сущностей
- **Минусы**:
  - Смешение доменов: серверная логика ботов переплетается с обработкой игроков
  - Клиент получает «ботовые» поля для игроков и «игроковые» поля для ботов
  - При расширении NPC-системы потребуется множество `isBot`/`isNPC` проверок
  - `World.movePlayer()` будет вызываться и для ботов, хотя у них другая модель движения

#### Вариант B (Выбран): Отдельная `bots` MapSchema

- **Описание**: Добавить `bots: MapSchema<ChunkBot>` в ChunkRoomState рядом с `players`.
- **Плюсы**:
  - Чёткое доменное разделение: игроки и NPC — разные коллекции
  - Клиент получает отдельные колбеки onAdd/onRemove для ботов — явная обработка
  - Будущие NPC-сущности (городские NPC, враги) легко добавляются как новые коллекции
  - Совместимо с архитектурой npc-service.md: NPCService оперирует собственным состоянием
  - ChunkBot-схема содержит только нужные NPC-поля (state, mapId), без sessionId
- **Минусы**:
  - Дополнительная MapSchema в патче (незначительный оверхед)
  - Дублирование кода рендеринга на клиенте (решается переиспользованием PlayerSprite)
  - Необходимость отдельных колбеков в PlayerManager для ботов

#### Вариант C: Отдельный Colyseus Room для ботов

- **Описание**: Боты живут в отдельной комнате, клиент подключается к двум комнатам.
- **Плюсы**:
  - Полная изоляция: бот-комната не влияет на основную
  - Масштабируется независимо
- **Минусы**:
  - Сложность: клиент подключается к двум комнатам одновременно
  - Физические столкновения между ботами и игроками невозможны из коробки
  - Двойная синхронизация позиций, дублирование состояния карты
  - Противоречит принципу Colyseus «одна комната = одно пространство»

### Обоснование

Вариант B выбран за чистое доменное разделение и прямой путь к интеграции с NPC-сервисом. Оверхед дополнительной MapSchema пренебрежимо мал (< 10 ботов). Переиспользование PlayerSprite на клиенте компенсирует кажущееся дублирование.

### Последствия

**Положительные**:
- ChunkRoomState имеет явное семантическое разделение: `players` (реальные пользователи), `bots` (AI-сущности)
- Клиентский код может обрабатывать ботов специфически (например, иконка NPC, курсор взаимодействия)
- При переходе к полноценной NPC-системе не потребуется миграция данных из `players`

**Отрицательные**:
- Два набора колбеков в PlayerManager (players + bots)

**Нейтральные**:
- ChunkBot — новый Colyseus Schema-класс со своими полями

## Решение 2: Размещение серверной логики ботов

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Разместить BotManager в `apps/server/src/npc-service/lifecycle/` для алигнмента с архитектурой npc-service.md |
| **Почему сейчас** | Первая NPC-логика задаёт структуру папок, которую будут использовать все последующие модули |
| **Почему это** | Прямое соответствие плановой архитектуре npc-service.md: lifecycle/ управляет спавном и тик-циклом |
| **Известные неизвестные** | Может ли BotManager остаться совместимым с NPCLifecycleManager из npc-service.md при добавлении AI-логики |
| **Kill criteria** | Если при внедрении M0.3 потребуется полный рефакторинг BotManager вместо расширения |

### Рассмотренные варианты

#### Вариант A: Внутри ChunkRoom

- **Описание**: Логика ботов прямо в `ChunkRoom.ts`.
- **Плюсы**: Простота, доступ к состоянию комнаты
- **Минусы**: Раздувание ChunkRoom (уже 440+ строк), смешение ответственностей, противоречит npc-service.md

#### Вариант B: В `apps/server/src/bots/`

- **Описание**: Отдельная директория `bots/` в сервере.
- **Плюсы**: Изоляция от основного кода
- **Минусы**: Создание параллельной структуры, которую позже придётся мигрировать в `npc-service/`

#### Вариант C (Выбран): В `apps/server/src/npc-service/lifecycle/`

- **Описание**: Размещение BotManager в подпапке lifecycle/ согласно структуре npc-service.md.
- **Плюсы**: Алигнмент с целевой архитектурой, нет будущей миграции, понятная иерархия
- **Минусы**: Создание npc-service/ до полной готовности NPC-системы

### Обоснование

npc-service.md определяет структуру `npc-service/lifecycle/NPCLifecycleManager.ts`. BotManager — это упрощённый предшественник NPCLifecycleManager. Размещение в той же директории обеспечивает нулевую стоимость миграции при развитии системы.

## Решение 3: Модель данных бота (Colyseus Schema)

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Создать класс `ChunkBot extends Schema` с полями: id, worldX, worldY, direction, skin, name, state, mapId |
| **Почему сейчас** | Определение модели данных — фундамент для всех последующих решений |
| **Почему это** | Поля покрывают текущие потребности (рендеринг, блуждание, взаимодействие) и расширяемы для NPC-системы |
| **Известные неизвестные** | Потребуется ли дополнительное поле `emotion` при интеграции с DialogueEngine |
| **Kill criteria** | Если добавление нового поля потребует миграции Colyseus-схемы для существующих клиентов |

### Рассмотренные варианты

#### Вариант A: Расширение ChunkPlayer дополнительными полями

- **Описание**: Добавить `isBot`, `state`, `mapId` в существующий ChunkPlayer.
- **Плюсы**: Один тип, единая обработка
- **Минусы**: Засоряет ChunkPlayer NPC-полями, которые не нужны игрокам

#### Вариант B: ChunkBot с минимумом полей (id, x, y, skin)

- **Описание**: Только визуальные данные, без state.
- **Плюсы**: Минимализм
- **Минусы**: Невозможно различить состояния бота (idle/walking) на клиенте, невозможно определить привязку к карте

#### Вариант C (Выбран): ChunkBot с полным набором полей

- **Описание**: `ChunkBot extends Schema` с id, worldX, worldY, direction, skin, name, state, mapId.
- **Плюсы**: Полная информация для клиента (анимация, имя, состояние), mapId для идентификации привязки к карте
- **Минусы**: Больше данных в патче (но все поля нужны)

### Обоснование

ChunkBot должен содержать достаточно данных для полноценного рендеринга и будущего расширения. Поле `state` (IDLE/WALKING) позволяет клиенту выбирать анимацию. `mapId` привязывает бота к карте хоумстеда (а не к пользователю напрямую), что семантически точнее: боты являются обитателями карты.

## Решение 4: Схема базы данных

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Создать таблицу `npc_bots` с Drizzle ORM в `packages/db/src/schema/`, с FK `mapId` на `maps.userId` (PK таблицы maps) |
| **Почему сейчас** | Персистентность — одно из основных требований PRD-009 |
| **Почему это** | Отдельная таблица с FK на maps.userId привязывает ботов к карте (а не к пользователю напрямую), что семантически точнее: боты — обитатели карты. Соответствует паттернам проекта (player_positions, maps) |
| **Известные неизвестные** | Потребуется ли в будущем миграция `npc_bots` в `npc_agents` из npc-service.md |
| **Kill criteria** | Если схема требует > 2 миграций для адаптации под полноценных NPC |

### Рассмотренные варианты

#### Вариант A: Хранение в Redis

- **Описание**: Ботовые данные в Redis с TTL.
- **Плюсы**: Быстрые чтение/запись
- **Минусы**: Redis не персистентен по умолчанию, нет связей с users, не соответствует паттерну проекта

#### Вариант B: Хранение в JSON-поле существующей таблицы

- **Описание**: Добавить JSON-колонку в таблицу `maps` или `player_positions`.
- **Плюсы**: Нет новой таблицы
- **Минусы**: Нарушение SRP таблицы, сложные запросы, невозможно индексировать отдельные поля ботов

#### Вариант C (Выбран): Отдельная таблица `npc_bots`

- **Описание**: `npc_bots` с uuid PK, FK `mapId` на `maps.userId` (PK карты), полями для имени, скина, позиции. Боты привязаны к карте, а не к пользователю напрямую.
- **Плюсы**: Чистая реляционная модель, индексируемость, соответствие паттерну проекта, семантически точная привязка к карте
- **Минусы**: Дополнительная таблица и миграция

### Обоснование

Проект использует Drizzle ORM с отдельными таблицами для каждой сущности (users, player_positions, maps, sprites, game_objects и т.д.). Таблица `npc_bots` следует этому паттерну. Привязка через `mapId` (FK на `maps.userId`) семантически точнее, чем прямая привязка к пользователю: боты — обитатели карты, которая принадлежит пользователю. Загрузка ботов происходит по `mapId` при входе на хоумстед. При эволюции в полноценных NPC можно добавить колонки (seed_persona, current_plan) или создать новую таблицу `npc_agents` с миграцией данных.

## Решение 5: Алгоритм блуждания

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Случайный выбор цели из проходимых тайлов + прямое движение с покадровой проверкой коллизий |
| **Почему сейчас** | Определяет основное видимое поведение бота |
| **Почему это** | Простота реализации, достаточная естественность для MVP, совместимость с существующей системой движения |
| **Известные неизвестные** | Достаточно ли выглядит «живым» прямое движение без A*-pathfinding |
| **Kill criteria** | Если > 30% тестировщиков отмечают, что боты «выглядят глупо» из-за застревания |

### Рассмотренные варианты

#### Вариант A: A* Pathfinding

- **Описание**: Полноценный A* для нахождения пути к случайной цели.
- **Плюсы**: Оптимальные пути, обход препятствий
- **Минусы**: Оверкилл для MVP, A* ещё не реализован в проекте, дополнительная зависимость

#### Вариант B (Выбран): Случайный выбор цели + прямое движение

- **Описание**: Каждые N тиков выбрать случайный проходимый тайл в радиусе. Двигаться к нему по прямой, проверяя проходимость каждого тика. При блокировке — выбрать новую цель.
- **Плюсы**: Простота, использует существующую `walkable[][]` сетку, быстрая реализация
- **Минусы**: Бот может выбирать цель за непроходимой стеной и «застревать»

#### Вариант C: Предопределённые маршруты (waypoints)

- **Описание**: Задать фиксированные точки маршрута для каждого хоумстеда.
- **Плюсы**: Предсказуемое поведение, нет застревания
- **Минусы**: Требует ручной настройки для каждого шаблона карты, не масштабируется

### Обоснование

Вариант B — оптимальный баланс между простотой и естественностью для MVP. Stuck-detection (выбор новой цели при 5 секундах без движения) компенсирует недостаток прямого движения. При переходе к M0.3 можно заменить на A* без изменения интерфейса BotManager.

## Решение 6: Рендеринг на клиенте

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Переиспользовать PlayerSprite для ботов, добавить колбеки для `bots` MapSchema в PlayerManager |
| **Почему сейчас** | Определяет клиентскую архитектуру отображения ботов |
| **Почему это** | Максимальное переиспользование кода, визуальная совместимость с игроками |
| **Известные неизвестные** | Потребуется ли BotSprite с дополнительной функциональностью (иконки состояния, пузырьки) |
| **Kill criteria** | Если PlayerSprite потребует > 5 bot-specific условий внутри своего кода |

### Рассмотренные варианты

#### Вариант A: Создать отдельный BotSprite-класс

- **Описание**: Новый класс, дублирующий логику PlayerSprite с бот-специфичными дополнениями.
- **Плюсы**: Полная свобода кастомизации
- **Минусы**: Дублирование ~170 строк кода (анимация, интерполяция, имя), нарушение DRY

#### Вариант B (Выбран): Переиспользование PlayerSprite

- **Описание**: Создавать `new PlayerSprite(...)` для ботов с теми же параметрами. Добавить отдельные onAdd/onChange/onRemove колбеки для `bots` в PlayerManager.
- **Плюсы**: Нулевое дублирование кода, боты визуально идентичны удалённым игрокам (те же анимации)
- **Минусы**: PlayerSprite не знает о том, что это бот (может потребоваться флаг в будущем)

#### Вариант C: Наследование (BotSprite extends PlayerSprite)

- **Описание**: BotSprite наследует PlayerSprite и переопределяет нужные методы.
- **Плюсы**: Расширяемость через наследование
- **Минусы**: Проект не использует наследование для спрайтов, усложнение для MVP

### Обоснование

PlayerSprite уже содержит всю необходимую логику: создание спрайта с скином, анимации, интерполяция, метка имени. Для MVP достаточно переиспользования as-is. В будущем можно добавить тонкую обёртку или флаг `isBot` для кастомизации (например, иконки NPC).

## Решение 7: Протокол взаимодействия

### Детали решения

| Пункт | Содержание |
|-------|-----------|
| **Решение** | Новые типы сообщений: `ClientMessage.NPC_INTERACT` и `ServerMessage.NPC_INTERACT_RESULT` |
| **Почему сейчас** | Определяет контракт клиент-сервер для взаимодействия с ботами |
| **Почему это** | Специфичный тип сообщений позволяет отличить NPC-взаимодействие от движения и других действий |
| **Известные неизвестные** | Будут ли в M0.3 использоваться те же типы сообщений или появятся dialogue_start/dialogue_message |
| **Kill criteria** | Если протокол потребует breaking change при добавлении диалогов |

### Рассмотренные варианты

#### Вариант A: Переиспользование существующих сообщений

- **Описание**: Использовать generic `ClientMessage.INTERACT` с полем `targetType`.
- **Плюсы**: Универсальность
- **Минусы**: Невозможно типизировать payload, смешение контекстов

#### Вариант B (Выбран): Специфичные NPC-сообщения

- **Описание**: `ClientMessage.NPC_INTERACT = 'npc_interact'`, `ServerMessage.NPC_INTERACT_RESULT = 'npc_interact_result'`.
- **Плюсы**: Чёткая типизация, явный контракт, легко расширяется (добавление `NPC_DIALOGUE_START` в M0.3)
- **Минусы**: Дополнительные типы сообщений (минимальный оверхед)

#### Вариант C: REST API вместо Colyseus-сообщений

- **Описание**: Клиент вызывает HTTP endpoint для взаимодействия с NPC.
- **Плюсы**: Независимость от Colyseus
- **Минусы**: Дополнительная задержка, нарушение единообразия (всё остальное через Colyseus-сообщения)

### Обоснование

Проект использует `ClientMessage`/`ServerMessage` enum-паттерн для всех клиент-серверных взаимодействий. NPC-взаимодействие должно следовать этому паттерну. В M0.3 `NPC_INTERACT_RESULT` может быть расширен полем `dialogueAvailable: true`, а новые сообщения `NPC_DIALOGUE_START`/`NPC_DIALOGUE_MESSAGE` добавлены без изменения существующих.

## Сравнительная матрица (все решения)

| Решение | Выбранный вариант | Главное преимущество | Главный компромисс |
|---------|-------------------|---------------------|-------------------|
| #1 Colyseus-состояние | Отдельная `bots` MapSchema | Чистое доменное разделение | Два набора колбеков на клиенте |
| #2 Размещение логики | `npc-service/lifecycle/` | Алигнмент с целевой архитектурой | Создание структуры до полной готовности |
| #3 Модель данных | ChunkBot с полным набором | Информация для рендеринга и расширения | Чуть больше данных в патче |
| #4 Схема БД | Таблица `npc_bots` (FK на maps) | Чистая реляционная модель, привязка к карте | Дополнительная миграция |
| #5 Алгоритм блуждания | Прямое движение + random target | Простота, быстрая реализация | Возможное застревание (mitigated) |
| #6 Рендеринг | Переиспользование PlayerSprite | Нулевое дублирование | Нет бот-специфичной визуализации |

## Руководство по имплементации

- Использовать Colyseus `@type` декораторы для ChunkBot аналогично ChunkPlayer
- Использовать dependency injection: BotManager получает walkability grid и db-клиент через конструктор
- Все бот-специфичные типы определять в `packages/shared` для переиспользования клиентом и сервером
- Придерживаться существующих паттернов: factory-функции для создания сущностей, сервис-функции для DB-операций
- Обработка ошибок: fail-fast для критичных операций (спавн), fire-and-forget с логированием для некритичных (сохранение позиции)
